## 1. Bug Fixes

### 1.1 `tform.bench.audio.melspectrogram` — Wrong signature
**Problem:** `bench/audio/mel.py` was a copy of the production `mel.py`, exposing `melspectrogram(signal: Tensor)` instead of the bench signature `melspectrogram(inputs: List, iterations, warmup)`.  
**Fix:** Rewrote bench orchestrator with correct signature and `BenchmarkReport` return.

### 1.2 `torch.stft` frame count mismatch
**Problem:** `torch.stft` defaults to `center=True`, padding the signal and producing more frames than the legacy CPU loop (`97 vs 101` for 1s audio). `_compute_fidelity` crashed with shape mismatch.  
**Fix:** Added `center=False` to all `torch.stft` calls to align frame count: `1 + floor((N − n_fft) / hop_length)`.

### 1.3 Float64 precision mismatch in mel legacy path
**Problem:** `_generate_mel_fb()` returns float64; `np.dot(float32_fft, float64_fb)` upcasts to float64, making the legacy more precise than the float32 MPS accelerated path. L∞ error exceeded the `1e-3` test threshold.  
**Fix:** Cached filter bank as `self._mel_fb_np = self._generate_mel_fb().astype(np.float32)`. Both paths now float32 throughout.

### 1.4 CQT kernel: `rfft` on complex input
**Problem:** `np.fft.rfft` requires real input. The CQT basis `window × exp(2πi·Q·n/N_k)` is complex.  
**Fix:** Replaced with `np.fft.fft(basis, n=n_fft)[:n_fft//2+1]` — full FFT sliced to one-sided spectrum.

### 1.5 CQT conjugation and normalization
**Problem:** CQT analysis requires `conj(K) @ X` (analysis direction), not `K @ X`. Additionally, the unnormalized FFT produced output magnitudes O(N_fft) instead of O(amplitude).  
**Fix:** Flipped cross-term signs in split-complex matmul; added `× (2/n_fft)` normalization to kernel.

### 1.6 MPS compatibility — `torch.mps.is_available` AttributeError
**Problem:** Older PyTorch versions (V100 machine, `acl2021_` env) have `torch.mps` as a module without `is_available`. Crashed with `AttributeError`.  
**Fix:** Added `hasattr(torch.mps, "is_available")` guard. Refactored into shared `detect_device()` in `src/tensorform/_device.py` — imported by all 12 affected files.

### 1.7 Python 3.8 compatibility — lowercase generic types
**Problem:** `tuple[np.ndarray, np.ndarray]` as return type annotation requires Python 3.9+. V100 runs Python 3.8.  
**Fix:** Added `from __future__ import annotations` to `audio/melt.py` and `bench/core.py`.

### 1.8 NFS symlink resolution
**Problem:** `Path.rglob("*.wav")` follows symlinks into NFS-mounted directories, returning paths like `/sessions/.../mnt/...` (container-internal paths not accessible on V100 host).  
**Fix:** Replaced `rglob` with `os.walk(followlinks=False)` in `bench/audio/_io.py`.

### 1.9 `BenchmarkReport` crash on empty trials
**Problem:** `BenchmarkReport([], [])` crashed with `IndexError` on `accel_trials[0]`.  
**Fix:** Added early `ValueError` with descriptive message.

### 1.10 False cosine similarity = 1.0 on degenerate input
**Problem:** `if norm_a and norm_b else 1.0` returned 1.0 when one norm was zero (catastrophic failure masked as perfect fidelity).  
**Fix:** Changed to `denom > 0.0`, returns `0.0` on degenerate vectors.

Previously duplicated in 13 files with 6-line if/elif/else blocks. Now imported everywhere.

### 2.3 HAC vs STFT+Mel distinction
Critical realization mid-session: `tform.audio.melspectrogram` implements **conventional STFT+Mel** — same as torchaudio. The library's core contribution is `tform.audio.melt` (Direct Mel Projection, no STFT). Compare scripts were updated to benchmark the right thing.

### 2.4 Split-complex arithmetic for MPS
MPS does not support complex tensor matmul. CQT and Gammatone use split-complex float32 arithmetic:
```
out_real = kr @ Xr + ki @ Xi   # conj(K) @ X, real part
out_imag = kr @ Xi - ki @ Xr   # conj(K) @ X, imag part
```

### 2.5 Benchmark shared core (`_bench_core.py`)
Identified ~60% code duplication across bench scripts. Extracted to `benchmarks/audio/_bench_core.py`:
- `DEVICE`, `POWER_W`, `MEDALS`
- `sync()`, `measure()`, `energy_mj()`, `gpu_sanity()`
- `format_report()`, `save_report()`

## 3. Operators Implemented

### 3.1 `tform.audio.melt` — MelT (Direct Mel Projection)
Reference: Camargo & Finger, *"MelT: GEMM-Native NDFT for Efficient Single-Stage Audio Frontends on Modern Accelerators"*, arXiv 2026.

**Equations:**
- W^(r)_{m,n} = w[n] · cos(2π f_m n / f_s)  — Eq. 8
- W^(i)_{m,n} = w[n] · sin(2π f_m n / f_s)  — Eq. 8
- R = X (W^(r))^T,  I = X (W^(i))^T          — Eq. 9
- S = R⊙R + I⊙I                               — Eq. 10
- MelT = log(S + ε)                            — Eq. 11

**Defaults (paper Table 3, spectral):** n_fft=400, hop=160, n_mels=80, f_min=80 Hz, f_max=7600 Hz

### 3.2 `tform.audio.mfcct` — MFCCT
**Equation:** C^MFCCT = M^MelT D^T = log(S + ε) D^T — Eq. 12

**Defaults (paper Table 3, cepstral):** n_fft=1200, hop=160, n_mels=128, K=13, f_min=0 Hz, f_max=8000 Hz

### 3.3 `tform.audio.cqt` — Constant-Q Transform
Frequency-domain precomputed kernels applied as split-complex matmul. Kernel: Hann-windowed complex sinusoid FFT, normalized by `2/n_fft`. n_fft auto-computed from Q and f_min (next power of 2).

### 3.4 `tform.audio.gammatone` — Gammatone Filterbank
4th-order gammatone impulse responses (ERB-spaced), precomputed in frequency domain. Split-complex matmul. Impulse responses computed in float64 for precision, cast to float32 before storage.

### 3.5 Conventional operators (existing)
- `tform.audio.melspectrogram` — HAC STFT + mel filterbank
- `tform.audio.mfcc` — HAC STFT + mel + DCT-II

## 4. Benchmark Infrastructure

### 4.1 File layout (final)
```
benchmarks/
├── audio/
│   ├── _bench_core.py       # shared utilities
│   ├── bench_mel.py         # MelT vs librosa / torchaudio / nnAudio
│   ├── bench_mfcc.py        # MFCCT vs librosa / torchaudio
│   ├── bench_cqt.py         # CQT vs librosa / nnAudio
│   ├── bench_gammatone.py   # Gammatone CPU vs GPU
│   └── run_all.py           # runs all 4 in sequence
└── tools/
    └── gen_dataset.py       # generates N×T-second WAV files from source
```

### 4.2 Report format
Each benchmark saves to `benchmarks/results/bench_{op}_{hostname}_{device}.txt` with:
- System environment (hostname, timestamps, timezone, OS, CPU, accelerator, TF/Python/PyTorch/NumPy versions)
- Dataset statistics (source, n_files, duration mean±SD, size mean±SD)
- Run configuration (all operator parameters)
- Latency ranking (1st/2nd/3rd with speedup vs 1st)
- Energy efficiency ranking (estimated mJ per call, power model: MPS=15W, CPU=45W, CUDA=NVML)

### 4.3 GPU sanity check
Before registering a contender as "GPU", the script runs 5 CPU iterations and 5 GPU iterations. If GPU is not >15% faster than CPU, the result is labeled `(CPU fallback)` — catching silent MPS fallbacks in torchaudio/nnAudio.

### 4.4 Energy model
```
E(mJ) = P(W) × t(s) × 1000
```
Power model: MPS=15W (Apple SoC TDP estimate), CPU=45W, CUDA=sampled via pynvml.

### 4.5 Reproducibility
- `os.walk(followlinks=False)` + `sorted()` — deterministic file order
- `torch.backends.cudnn.benchmark = False` — fixed kernel selection on CUDA
- `torch.backends.cudnn.deterministic = True`
- Warmup iterations before timing

## 6. Key Insights

**MelT/MFCCT win everywhere.** Eliminating the STFT reduces kernel launch overhead, intermediate tensor allocation, and memory bandwidth. On MPS the advantage is amplified because competitors don't support MPS natively.

**CQT — honest finding.** nnAudio's 1D convolution is 1.1–1.7x faster than TF for long signals (160s) on every GPU platform tested (MPS, H100, V100). This is consistent and should be reported as-is. TF still wins by 7–98x vs librosa (CPU). The advantage of TF's static matmul approach is larger for short signals and on backends where nnAudio's conv falls back to CPU.

**Gammatone — most dramatic.** The CPU legacy is inherently sequential (frame-by-frame complex matmul). The GPU batches all frames into one matmul call. 1600x speedup on V100 is real physics, not arithmetic — the operation maps perfectly onto tensor units.

**Apple Silicon specificity.** The largest competitive advantage is on MPS where torchaudio and nnAudio frequently fall back to CPU. TensorForm's GEMM-native design is backend-agnostic and runs natively on MPS, CUDA, and CPU with a single code path.

**Datasets must be comparable.** CelebVox (~7s) and Libri6000_100_160 (160s) give incomparable absolute speedups vs librosa. Re-run MPS benchmarks with 160s files for fair cross-platform comparison.
